void paint(Canvas canvas, Size size) {
  _zonePaths.clear();
  _drawGrid(canvas, size);          // subtle grey alignment grid
  for (final zone in zones) {
    _drawZone(canvas, size, zone);  // polygon fill + stroke per zone
  }
  // Vertex drag handles on selected zone only
  if (selectedZoneId != null) {
    final selected = zones.where((z) => z.id == selectedZoneId).firstOrNull;
    if (selected != null) _drawVertexHandles(canvas, size, selected);
  }

3. Changes from the Proposal

Area Proposed Implemented Reason
Local Storage Hive (key-value store) Drift (SQLite ORM) App data is highly relational: zones contain fixtures, planograms reference products, proposals link to planograms. Hive's flat key-value model would have required manual join logic. Drift provides typed tables, reactive queries, and foreign-key support out of the box — a better fit discovered while designing the schema.
User Roles 2 roles: coordinator, staff 3 roles: coordinator, manager, staff During implementation it became clear that store managers need their own operational authority (approve/reject photo docs and planogram proposals) without full coordinator privileges. A separate manager role cleanly separates these concerns and makes the permission model realistic for actual retail environments.
Multi-Store Layer Single-store assumed Full multi-store: create, invite-code join, approval flow, store groups, store switcher A retail VM app is meaningless without a store context. The proposal assumed this implicitly; building it explicitly required adding 5 new Drift tables (stores, store_memberships, store_groups, store_group_members, planogram_proposals) and 7 new screens. This expansion makes the app substantially more realistic and portfolio-worthy.
Planogram Workflow Publish/draft states only Full proposal system: staff submit changes, manager/coordinator approve/reject Discovered during implementation that read-only planogram display does not demonstrate meaningful Flutter complexity. Adding the proposal/approval workflow exercises async state management, role guards, and multi-user data flows — closer to the "~5 assignments" workload target.
Weeks 5–6 (skipped in proposal) Timeline gap in proposal v0.2 design spec and agent-based parallel implementation Weeks 5–6 were used to design the v0.2 architecture and begin parallel feature implementation across the multi-store layer, zone redesign, and role enforcement modules.

5. Difficulties & Plan to Overcome Them

Difficulty 1 — Drift Code Generation Complexity

Drift generates Companion types, Data classes, and DAO method stubs into app_database.g.dart, not into individual table files. Screens that need StoresTableCompanion must import app_database.dart, not stores_table.dart — a non-obvious pattern that caused a wave of "undefined class" compile errors across all feature screens when the schema was first wired up.

Resolution: Established a project-wide rule (documented in the decision log): all screens import app_database.dart directly. Applied consistently across all DAOs and providers. No further occurrences.

Difficulty 2 — Riverpod Generator Deprecations

riverpod_generator ^2.4.3 generates typed *Ref classes (e.g., ActiveStoreRef) that are deprecated by riverpod_annotation ^2.3.5 in favor of a plain Ref type. These appear as warnings across all generated provider files and cannot be resolved without a breaking package upgrade.

Plan: Pin both packages at current versions for the remainder of the project to keep generated code stable. Monitor riverpod_generator for a stable release resolving the Ref type change; upgrade after final submission if time allows.

Difficulty 3 — RoleGuard + Flutter Layout Constraints

Flutter's Expanded widget must be a direct child of a Row/Column/Flex. Wrapping an Expanded inside RoleGuard (which returns a SizedBox.shrink() fallback) caused layout assertion errors because the Expanded was no longer a direct Flex child.

Resolution: Moved RoleGuard to wrap the outer Padding containing the entire button row. The fallback collapses the whole button bar rather than individual buttons — cleaner UX and correct layout hierarchy.

Difficulty 4 — CustomPainter Hit Testing for Polygons

Zone polygons are arbitrary shapes; Flutter's standard tap detection works on bounding boxes, not polygon paths. Initial implementation triggered tap events on zones that weren't actually tapped.

Resolution: Cache each zone's Path object in a Map<String, Path> during paint(), then use path.contains(localPosition) in the GestureDetector's onTapDown handler to do accurate point-in-polygon hit testing. This pattern is now documented and reused in the Floor Builder canvas.
