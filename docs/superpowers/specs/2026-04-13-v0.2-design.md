# Merch Mobile v0.2 Design Spec

**Date:** 2026-04-13  
**Author:** Van Robbins + Claude  
**Status:** Approved

---

## Overview

v0.2 builds real functionality on top of the v0.1 scaffold. The primary goals are:

1. Wire up four existing stub features to the Drift database
2. Add multi-store support with invite-code-based joining and coordinator approval
3. Add store groups (many-to-many region groupings)
4. Add a role-aware Dashboard home screen
5. Add product → planogram slot assignment
6. Fully redesign the Zone Manager with editable polygon shapes and zone types
7. Enforce role-based permissions consistently across all screens

**Approach:** Schema-first (Agent 1 ships the DB migration and store layer; Agents 2–5 build in parallel on top; Agents 6–7 polish and test in Phase 2).

## 1. Database Schema Changes

### New Tables

#### `stores`
```
id            TEXT PRIMARY KEY
name          TEXT NOT NULL
invite_code   TEXT UNIQUE NOT NULL   -- 6-char uppercase alphanumeric (e.g. AX7K2P), generated via dart:math Random seeded from store UUID entropy
created_at    INTEGER NOT NULL
owner_uid     TEXT NOT NULL          -- Firebase UID of the creator (coordinator)
```

#### `store_memberships`
```
id            TEXT PRIMARY KEY
store_id      TEXT NOT NULL REFERENCES stores(id)
user_uid      TEXT NOT NULL          -- Firebase UID
role          TEXT NOT NULL          -- coordinator | manager | staff
display_name  TEXT NOT NULL
status        TEXT NOT NULL          -- pending | active | rejected
joined_at     INTEGER NOT NULL
```

#### `store_groups`
```
id              TEXT PRIMARY KEY
name            TEXT NOT NULL
description     TEXT
created_by_uid  TEXT NOT NULL
created_at      INTEGER NOT NULL
```

#### `store_group_members`
```
id            TEXT PRIMARY KEY
group_id      TEXT NOT NULL REFERENCES store_groups(id)
store_id      TEXT NOT NULL REFERENCES stores(id)
added_at      INTEGER NOT NULL
added_by_uid  TEXT NOT NULL
```
A store can belong to many groups; a group can contain many stores.

#### `planogram_proposals`
```
id                TEXT PRIMARY KEY
planogram_id      TEXT NOT NULL REFERENCES planograms(id)
store_id          TEXT NOT NULL REFERENCES stores(id)
proposed_by_uid   TEXT NOT NULL
proposed_at       INTEGER NOT NULL
status            TEXT NOT NULL    -- pending | approved | rejected
notes             TEXT
slot_changes      TEXT             -- JSON diff of slot changes
reviewed_by_uid   TEXT
reviewed_at       INTEGER
```

### Modified Tables (existing)

All five existing tables gain `store_id TEXT NOT NULL REFERENCES stores(id)` via a Drift schema migration (schemaVersion bump + `MigrationStrategy`).

| Table | Additional changes |
|---|---|
| `zones` | + `shape_points TEXT` (JSON array of `{x,y}` vertices), + `zone_type TEXT` (display\|cash_wrap\|fitting_room\|entrance\|storage\|other), + `notes TEXT` |
| `fixtures` | + `planogram_id TEXT` (nullable FK → planograms.id) |
| `products` | store_id only |
| `planograms` | store_id only |
| `photo_docs` | store_id only |

### Active Store Provider

`activeStoreProvider` (Riverpod) holds the currently selected store ID, persisted to SharedPreferences across launches. All DAOs gain a `watchByStore(storeId)` method. Every screen scopes its queries to the active store reactively. A `currentMembershipProvider` exposes the current user's role within the active store.

## 2. Store Onboarding & Multi-Store Flow

### First Launch (no store membership)

A **StoreGateScreen** intercepts unauthenticated-but-logged-in users with no active membership. Two options:

**Create a Store:**
- Enter store name + optional location
- Server auto-generates a 6-char invite code
- Creator becomes coordinator automatically
- Post-creation: show invite code with Copy + Regenerate options

**Join with Invite Code:**
- Enter 6-char code
- On valid code: creates a `store_memberships` row with `status = pending`, `role = staff`
- User sees a **PendingApprovalScreen**: "Waiting for coordinator/manager approval"
- User is locked out of the store until approved

### Approval Flow

**Who can approve:** coordinator or manager  
**Who can assign roles on approval:**
- Coordinator: staff or manager
- Manager: staff only (coordinator controls manager promotions)

Coordinator/manager see a **Members Screen** with:
- "Pending Requests" section with a badge count
- Each pending member has Approve (→ role picker) and Deny buttons
- "Active Members" section with Edit Role option (coordinator only for manager role)

On approval: `status` flips to `active`, role is set. User's pending screen auto-navigates to the app.

### Multi-Store Switcher

The app header shows the active store name + current user's role. Tap to open a **StoreSwitcherSheet** listing:
- All stores the user is an active member of (with role label)
- Active store highlighted
- "Join Another Store" shortcut at the bottom

Switching stores updates `activeStoreProvider` — all screens re-query instantly.

### Store Groups

Coordinators can create groups (e.g., "Gap Region 1 — East Coast") and add/remove stores. A **GroupManagementScreen** lists all groups for the active store and allows CRUD. Groups appear in the store switcher sheet for context. Group membership is informational in v0.2 (no cross-store analytics yet).

## 3. Role Permissions

| Feature | Coordinator | Manager | Staff |
|---|---|---|---|
| Store — create/invite/approve | Full (any role) | Approve/deny (staff only) | View own membership |
| Store Groups | Full CRUD | View only | View only |
| Zone Manager | Full CRUD | Full CRUD | View only |
| Floor Builder | Full CRUD | Full CRUD | View only |
| Product Catalog | Full CRUD | Full CRUD | Browse/search |
| Planograms | Full CRUD | Full CRUD + approve proposals | Propose changes only |
| Planogram Proposals | Approve/reject | Approve/reject | Submit + view own status |
| Photo Docs | Full | Approve/reject | Capture + view own |
| Dashboard | All stats | Planogram/photo stats + pending | Personal stats |
| Auto Build | Full | Full | View only |

A `RoleGuard` widget wraps any edit control and hides it if the current user's role is insufficient. Staff attempting restricted actions via navigation see an "Access restricted" snackbar.

## 4. Zone Manager Redesign

### Zone Map Canvas

Zones are rendered as arbitrary polygons using `CustomPaint`. In coordinator/manager edit mode:
- Vertex handles (draggable circles) appear on each corner
- Dragging a vertex reshapes the polygon in real-time and persists on release
- "Edit Vertices" vs "Replace Shape" toggle in the properties panel

### Zone Shape Picker

When adding a zone or replacing its shape, a bottom sheet offers predefined shapes:
- Rectangle, L-shape, T-shape, U-shape, diagonal cut
- After selection, coordinator can switch to vertex editing for fine-tuning

### Zone Types

Each zone has a `zone_type`:
- **Display** — has fixtures and planograms; tapping opens the Zone Detail view
- **Cash Wrap, Fitting Room, Entrance, Storage, Other** — blank zones; tapping opens only the Properties panel (name, color, shape editing). No interior view, no fixtures, no planograms.

Non-display zones are rendered with a dashed stroke on the map to distinguish them visually.

### Zone Detail View (Display zones only)

Shows the zone's interior floor plan with:
- Fixtures rendered as labelled rectangles
- Planogram badge on each fixture that has a `planogram_id` assigned
- Grey "Unassigned" chip on fixtures with no planogram
- "Assign Planogram" FAB for coordinator/manager
- Staff see read-only view

### Schema additions
- `zones.shape_points` replaces fixed width/height
- `zones.zone_type` determines display vs blank behavior
- `fixtures.planogram_id` FK fixes the broken planogram ↔ zone linkage

## 5. Feature Wiring

### Floor Builder
- Fixtures save/load from `FixturesDAO` scoped to `(storeId, zoneId)`
- Add/move/rotate/delete all persist immediately on action
- `planogram_id` FK on fixture wired to zone detail planogram badges
- Staff see read-only canvas (edit controls hidden via RoleGuard)

### Product Catalog
- Real CRUD via `ProductsDAO` scoped to active store
- Coordinator/manager: add, edit, delete products via `ProductFormScreen`
- Staff: browse + search only
- Product count surfaced on Dashboard

### Planogram Editor
- Slot editing persists via `PlanogramsDAO` + `PlanogramSlotsDAO`
- **Product assignment:** slots show "Assign Product" button → opens `ProductSlotPicker` (searchable sheet from catalog) → stores `planogram_slots.product_id`
- **Staff proposals:** staff see "Propose Change" instead of direct edit → opens `PlanogramProposalScreen` → creates row in `planogram_proposals`
- **Review queue:** `ProposalReviewScreen` for manager/coordinator — shows diff, approve/reject with optional notes

## 6. New Features

### Dashboard (home tab)

Role-aware stat cards replacing the current empty home tab:

| Role | Stats shown |
|---|---|
| Coordinator | Zone count, fixture count, product count, pending join requests, pending proposals, pending photo approvals |
| Manager | Product count, pending proposals, pending photo approvals |
| Staff | Own photo submission count + status, own proposal count + status |

### Product → Planogram Slot Assignment

In `PlanogramDetailScreen`:
- Each slot renders assigned product name + SKU, or an "Assign Product" placeholder
- Tapping opens `ProductSlotPicker` — a searchable bottom sheet from the store's product catalog
- Coordinator/manager assign directly; staff submit as a proposal

## 7. Agent Division

### Phase 1 — Functional Implementation

**Ground rules for all Phase 1 agents:** Ship working functionality. Default Flutter widgets are acceptable. Do not spend time on polish, animations, or minor bugs. Flag UI issues with `// TODO(Agent6)` comments and move on.

#### Agent 1 — Foundation & Schema *(GATE — runs first)*
- New tables: stores, store_memberships, store_groups, store_group_members, planogram_proposals
- New DAOs: stores, store_memberships, store_groups, planogram_proposals
- Modified: app_database.dart (migration), all existing tables (store_id FK + zone schema changes), @freezed models, providers (activeStoreProvider, currentMembershipProvider), router (store gate + members routes)
- New feature: `lib/features/store/` — StoreGateScreen, CreateStoreScreen, JoinStoreScreen, PendingApprovalScreen, StoreSwitcherSheet, MembersScreen, GroupManagementScreen
- **Also:** `lib/core/widgets/role_guard.dart` — RoleGuard widget (checks currentMembershipProvider; Agents 2–5 use this immediately)

#### Agent 2 — Zone Manager Redesign *(parallel after Agent 1)*
- Polygon canvas with vertex drag handles
- Zone shape picker (predefined shapes)
- Zone type selector (display vs blank)
- Zone detail screen (display zones only: fixtures + planogram badges)
- ZonesDAO: watchByStore, shape_points, zone_type support

#### Agent 3 — Floor Builder + Product Catalog *(parallel after Agent 1)*
- Floor Builder: fixtures persist to FixturesDAO (store-scoped), planogram_id FK wired
- Product Catalog: real CRUD from ProductsDAO, ProductFormScreen for add/edit

#### Agent 4 — Planogram Editor + Dashboard *(parallel after Agent 1)*
- Planogram detail: real slot editing, ProductSlotPicker, staff proposal button
- PlanogramProposalScreen + ProposalReviewScreen
- DashboardScreen + DashboardProvider (role-aware stats)

#### Agent 5 — Role Enforcement *(parallel after Agent 1)*
- route_guards.dart: store gate guard, role-based route blocking
- Audit pass over v0.1 screens (auth, photo_docs, auto_build) — apply RoleGuard where missing
- Agents 2/3/4 are responsible for RoleGuard on their own new screens; Agent 5 audits and fills gaps after merge
- No UI polish; flag issues with TODO(Agent6)

### Phase 2 — Polish & Testing *(after all Phase 1 agents merge)*

## 8. Out of Scope for v0.2

- Cross-group analytics / multi-store reporting
- Firebase sync for new tables (SyncService extension)
- Photo doc approval workflow (approved design, deferred)
- Auto Build AI logic (screen exists, logic remains stub)
- Push notifications for approvals
