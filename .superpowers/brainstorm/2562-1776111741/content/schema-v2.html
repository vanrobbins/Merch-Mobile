Design Section 1 — Database Schema (updated)

New tables + store_id on all existing tables + store groups

New Tables

stores

id TEXT PRIMARY KEY
name TEXT NOT NULL
invite_code TEXT UNIQUE NOT NULL
created_at INTEGER NOT NULL
owner_uid TEXT NOT NULL

store_memberships

id TEXT PRIMARY KEY
store_id TEXT → stores.id
user_uid TEXT NOT NULL
role TEXT coordinator | manager | staff
display_name TEXT NOT NULL
joined_at INTEGER NOT NULL

store_groups

id TEXT PRIMARY KEY
name TEXT NOT NULL
description TEXT
created_by_uid TEXT NOT NULL
created_at INTEGER NOT NULL

e.g. "Gap Region 1 — East Coast"

store_group_members

id TEXT PRIMARY KEY
group_id TEXT → store_groups.id
store_id TEXT → stores.id
added_at INTEGER NOT NULL
added_by_uid TEXT NOT NULL

A store can belong to many groups.
A group can contain many stores.

planogram_proposals

id TEXT PRIMARY KEY
planogram_id TEXT → planograms.id
store_id TEXT → stores.id
proposed_by_uid TEXT NOT NULL
proposed_at INTEGER NOT NULL
status TEXT pending | approved | rejected
notes TEXT
slot_changes TEXT — JSON diff of slots
reviewed_by_uid TEXT
reviewed_at INTEGER

Existing Tables — store_id FK added

Every existing table gets store_id TEXT NOT NULL REFERENCES stores(id) via Drift schema migration (schemaVersion bump).

zones fixtures products planograms photo_docs

Active Store in App State

activeStoreProvider holds the currently selected store (persisted to SharedPreferences across launches).
All DAOs gain watchByStore(storeId). Every screen scopes to the active store reactively.
A store switcher in the app header lets users with multiple memberships jump between stores.

Who can manage groups?

Only coordinators can create groups and add/remove stores from them.
Group membership is informational in v0.2 — used for display/filtering. Cross-group analytics are a future feature.
